ADR-006: Playmaker-style FSM System

2026-05-27 Adopted

Introduced a pure data-driven finite state machine (FSM) system inspired by Unity Playmaker. The design strictly follows ECS principles: no callbacks, no OOP inheritance in state logic, communication via event components only.

Core Architecture

Design Principles

  • Pure data components — FSM state is stored in components with no behavior
  • Event-driven communication — systems communicate via event components, not callbacks
  • Single-responsibility systems — each system does one thing (sense, decide, or act)
  • Request/Event pattern — sensors emit events, FSM processes them, action systems respond

FSM Components (Engine Layer)

FSMStateComponent

struct FSMStateComponent {
    uint32_t currentState = 0;  // Index into FSMDefinition::stateNames
    float stateTime = 0.0f;     // Time in current state
    float totalTime = 0.0f;     // Total FSM runtime
};

FSMDefinition

struct FSMDefinition {
    std::vector<std::string> stateNames;
    std::vector<FSMTransitionRule> eventTransitions;
    std::vector<FSMTimedTransition> timedTransitions;
    uint32_t initialState = 0;
};

FSMStateChangeEvent

Emitted by FSMSystem::update when a state transition occurs. Game systems read this component to trigger effects.

struct FSMStateChangeEvent {
    uint32_t fromState;
    uint32_t toState;
    float previousStateTime;
};

Builder API

auto def = FSMDefinitionBuilder()
    .state("Idle")
    .state("WaitingForVisible")
    .state("Displaying")
    .state("WaitingForHidden")
    .transition("Idle", "WaitingForVisible", "ShowSubtitle")
    .transition("WaitingForVisible", "Displaying", "PageVisible")
    .transition("Displaying", "WaitingForHidden", "HideSubtitle")
    .transition("WaitingForHidden", "Idle", "PageHidden")
    .initialState("Idle")
    .build();

Migration: Existing Components

TriggerVolumeComponent — Callbacks to Events

Before: callbacks stored directly in component

// OLD - OOP style with callbacks
struct TriggerVolumeComponent {
    TriggerHandle trigger;
    std::function<void(Entity, Entity)> onEnter;
    std::function<void(Entity, Entity)> onExit;
};

After: pure data component + separate event components

// NEW - Pure data + event pattern
struct TriggerVolumeComponent {
    TriggerHandle trigger;
};

struct TriggerEnterEvent {
    entt::entity triggerEntity;
    entt::entity otherEntity;
};

struct TriggerExitEvent {
    entt::entity triggerEntity;
    entt::entity otherEntity;
};

Game systems now react to TriggerEnterEvent/TriggerExitEvent instead of having callbacks invoked.

State Constants

Hand-written enums converted to namespaced constexpr uint32_t for FSM state indices:

// Game layer (Vaporware/src/components.hpp)
namespace SubtitleStates {
    constexpr uint32_t Idle = 0;
    constexpr uint32_t WaitingForVisible = 1;
    constexpr uint32_t Displaying = 2;
    constexpr uint32_t WaitingForHidden = 3;
}

Migration: Monolithic to Single-Responsibility Systems

SubtitleQueueSystem Split

The original SubtitleQueueSystem::update() handled input detection, page animation sensing, timer management, and page actions in one function. Split into four focused systems:

SystemResponsibilityEmits/Consumes
SubtitleInputSystemDetect advance request→ "ShowSubtitle" event
SubtitlePageSensorSystemMonitor page animation state→ "PageVisible"/"PageHidden"
SubtitleTimerSystemUpdate display timer→ "HideSubtitle" event
SubtitleActionSystemRespond to state changes← FSMStateChangeEvent

Execution Order

// main.cpp game loop
SubtitleInputSystem::update(registry);
SubtitlePageSensorSystem::update(registry);
SubtitleTimerSystem::update(registry, deltaTime);
Vapor::FSMSystem::update(registry, deltaTime);  // Process events, emit FSMStateChangeEvent
SubtitleActionSystem::update(registry);        // React to state changes

UI Page System Relationship

Page architecture unchanged. The FSM system complements the existing Page architecture — it doesn't replace it.

The UI system uses a hybrid approach:

  • Page subclasses (e.g., SubtitlePage, ChapterTitlePage) own behavior: element references, animation state, fade timers
  • ECS components control when pages show/hide: UIVisibleTag, SubtitleQueueComponent
  • FSM manages the state machine logic for complex sequences (show → wait for visible → display → wait for hidden → idle)

        .state("Displaying")
        .state("WaitingForHidden")
        .transition("Idle", "WaitingForVisible", "ShowSubtitle")
        .transition("WaitingForVisible", "Displaying", "PageVisible")
        .transition("Displaying", "WaitingForHidden", "HideSubtitle")
        .transition("WaitingForHidden", "Idle", "PageHidden")
        .initialState("Idle")
        .build();
}

Why This Design

Rejected: Callback-based FSM

Initial implementation used std::function callbacks for onEnter/onExit. Rejected because:

  • Breaks pure-data ECS principle
  • Callbacks capture mutable state, making serialization impossible
  • Testing requires mocking function objects

Rejected: Request components in engine layer

Game-specific requests (e.g., ShowSubtitleRequest) belong in game layer, not engine. Engine provides generic FSMEventQueue + FSMStateChangeEvent; game decides what those mean.

Chosen: Event-driven single-responsibility systems

  • Each system does exactly one thing — easy to test, debug, reorder
  • Communication is explicit via event components
  • No hidden state transitions in callbacks
  • FSM logic is visible in data, not buried in code
