ADR-003: Test Isolation Strategy

2026-04-29 Adopted

Context

Initializing the Vapor library requires a graphics device (Metal/Vulkan context), a physics engine (Jolt, requiring SSE/AVX CPU features), and an audio device (miniaudio). None of these are available on headless CI runners.

If test binaries link the full Vapor library, any #include <Vapor/scene.hpp> pulls in Physics3D, CharacterController, and VehicleController declarations — and their implementations require Jolt dynamic linking, even if the test never calls a physics function.

Decision

Test binaries do not link the Vapor CMake target. Instead:

  • Directly #include pure header logic (e.g., action_manager.hpp, camera.hpp) where the implementation is fully inline
  • Use stubs or partial reimplementations for classes with external dependencies (e.g., a MinimalNode stub replicating only the pure logic of Node)
  • Link only: Catch2::Catch2WithMain, glm::glm, fmt::fmt

Rationale

This strategy allows tests to run on any CI runner without GPU or audio hardware, complete in milliseconds, and test only the logic under examination — not the entire subsystem stack.

Trade-offs

Stubs must precisely replicate the source logic. Any drift between a stub and its source removes the characterization guarantee. Stub synchronization is currently enforced by code review.
  • When Node or other stubbed classes change, stubs must be updated manually
  • Integration tests (real ResourceManager loads, real Physics3D simulation) require a separate environment with appropriate hardware

Future Work

  • For Physics3D testing: consider a dedicated integration test suite on a GPU-capable runner, or verify Jolt's headless mode (CPU-only, no GPU required)
