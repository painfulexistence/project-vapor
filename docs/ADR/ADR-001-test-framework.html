ADR-001: Test Framework Selection

2026-04-29 Adopted

Context

Project Vapor had no automated tests. The demo application (Vaporware/src/main.cpp, ~25k LOC) was the only validation mechanism. Any refactor had no safety net; correctness could only be verified by manually building and running the demo.

The Phoenix Workflow Phase 2 required characterization tests for core engine paths.

Decision

Adopt Catch2 v3 as the test framework.

Options Considered

OptionProsCons
GoogleTestWidely used, strong fixture supportExtra install step, slightly more complex CMake integration
Catch2 v3Simple CMake integration, catch_discover_tests for automatic CTest registration, available via vcpkgSlightly different fixture syntax
Hand-rolled assertionsZero dependenciesNo test report, no CI integration

Rationale

  • vcpkg.json requires a single "catch2" entry
  • Catch2::Catch2WithMain target eliminates hand-written main()
  • catch_discover_tests() makes each TEST_CASE a separate CTest entry automatically
  • Test binaries run standalone without CTest

Constraints

  • Test targets do not link the Vapor library — avoids triggering Metal/Vulkan/Jolt/miniaudio initialization
  • GPU-dependent systems are tested through stubs or not tested in CI
  • Test binaries must run on headless GitHub Actions runners (macos-15)

Consequences

  • vcpkg.json gains a "catch2" dependency
  • Root CMakeLists.txt gains enable_testing() and add_subdirectory(tests)
  • tests/CMakeLists.txt defines independent test executables
  • CMakePresets.json gains a ci preset for GitHub Actions
