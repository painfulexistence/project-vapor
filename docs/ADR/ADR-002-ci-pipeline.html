ADR-002: CI Pipeline Design

2026-04-29 Adopted

Context

No CI/CD pipeline existed. Merges had no automatic verification — breaking changes were only caught when someone manually built and ran the demo.

Decision

Create a two-job pipeline in .github/workflows/ci.yml:

  • test — compile and run unit test binaries only (no GPU, no demo app)
  • build-check — runs after test passes; compiles the full Vapor library

Rationale

Why two jobs?

Unit tests are much faster than a full build — test binaries have no Metal/Vulkan shader compilation. Putting tests first means logic errors are caught before waiting for the full build.

Why macos-15?

The primary development platform is macOS (Metal backend). The macos-15 runner uses Apple Silicon (M1), matching the development environment. Linux/Vulkan CI requires a software renderer (SwiftShader/lavapipe) and is deferred.

Why ccache?

Metal shader compilation and vcpkg dependency builds are slow. ccache with actions/cache persists the compiler cache across runs, significantly reducing CI time on subsequent pushes.

Why exclude Vaporware?

main.cpp requires a window system and Metal context. Neither is available on headless CI runners. CI compiles only to the Vapor library boundary.

Constraints

  • No demo execution in CI
  • No GPU-dependent tests (Renderer, Physics3D require a device)
  • Vulkan CI deferred pending SwiftShader/lavapipe verification

Future Work

  • Linux runner for Vulkan backend build check
  • clang-tidy / clang-format lint job
  • Address sanitizer run (-fsanitize=address exists in dev preset but not CI)

Consequences

  • .github/workflows/ci.yml created
  • CMakePresets.json gains a ci preset inheriting global-vcpkg, using $VCPKG_ROOT env var
