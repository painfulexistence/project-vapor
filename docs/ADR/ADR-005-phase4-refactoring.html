ADR-005: Phase 4 Refactoring

2026-05-01 Adopted

Phase 4 targeted surgical cleanup of the highest-value technical debt in the demo layer (Vaporware/), without touching engine internals that lack test coverage. Four changes were made: T1 (PhysicsHandle type safety), T2 (SceneBuilder extraction), T3 (ECS systems naming), T4 (RmlUI helpers).

T1 — PhysicsHandle<Tag> Template

Decision

Replace the two structurally identical BodyHandle/TriggerHandle bare structs with a tag-template:

template<typename Tag>
struct PhysicsHandle {
    Uint32 rid = UINT32_MAX;
    bool valid() const { return rid != UINT32_MAX; }
    bool operator==(const PhysicsHandle&) const = default;
};
using BodyHandle    = PhysicsHandle<BodyTag>;
using TriggerHandle = PhysicsHandle<TriggerTag>;

Rationale

The original bare structs were structurally identical — physics.destroyBody(triggerHandle) compiled without error. The template makes accidental cross-type assignment a compile error at zero runtime cost. This mirrors the GPUHandle<Tag> pattern from ADR-004.

Impact

Pure type-alias change. All call sites keep the same names and no logic changes. Forward declarations of BodyHandle as a struct had to be replaced with full includes, since type aliases cannot be forward-declared.

T2 — SceneBuilder Extraction

Decision

Extract the 195-LOC entity-creation block from main() into Vaporware/src/scene_builder.hpp as a single buildScene(...) free function returning a SceneResources struct.

Why a free function, not a class?

The scene builder has no persistent state — it runs once, populates the registry, and returns handles. A class would add boilerplate with no benefit. The SceneResources return struct carries the two cross-cutting handles main() needs: cube1 (referenced by FollowCameraComponent::target) and global (the singleton entity for switch requests).

Impact

main() reduced from 627 to 435 lines. No new dependencies; scene_builder.hpp includes only headers already present in main.cpp.

T3 — ECS Systems Naming Normalization

Decision

Convert all 8 free functions in systems.hpp from updateXSystem(...) to XSystem::update(...) static-class form, matching the 3 classes already present.

Rationale

The mixed convention required two grep patterns to find all systems. A single *System::update pattern now covers everything. Static-class form also makes it easier to add per-system state later without changing call sites.

Impact

Pure rename. No logic changes. All call sites in main.cpp updated.

T4 — RmlUI Helper Functions

Decision

Extract two helpers into namespace RmlUIHelpers at the top of systems.hpp:

  • ensureDocument(docPtr, rml, path) — loads a document once, returns the pointer
  • tickTimer(timer, dt, duration) — advances a timer and returns true when it fires

Why not a base class?

The 5 RmlUI systems (HUD, ScrollText, Letterbox, Subtitle, ChapterTitle) have different state enums, different element lookups, and different transition logic. A base class would require virtual dispatch and shared state that doesn't cleanly unify. Two free helpers eliminate ~60 lines of copy-paste without introducing coupling.

tickTimer was applied only where the pattern was a simple timer += dt; if (timer >= duration) with no interrupt check between them. Interrupt-checked states were left as-is to avoid semantic drift.

Known Remaining Debt

ECS / SceneGraph dual model: Vaporware uses both EnTT components and Scene nodes for the same objects. Unification would require touching engine internals without test coverage. Deferred.

Vulkan backend parity: 28 Metal render passes have no Vulkan counterpart. Tracked separately. Deferred to a dedicated graphics sprint.
