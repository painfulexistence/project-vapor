ADR-008: Particle ECS Integration — CPU Spawn Logic, Per-Emitter GPU Slots, and Force Field Architecture

2026-05-30 Adopted Branch: claude/entity-particle-emitter-gpu-RrV6m

This record covers the design and implementation of ECS-driven particle emission in Project Vapor — connecting the entt::registry entity layer to the existing GPU compute-based particle simulation — and the sequence of decisions made along the way. It is written in devlog style: roughly in the order things were discovered, not in the order a polished spec would present them.

Context — The Situation Before

The Vulkan and Metal renderers already had a working GPU particle system: a global fixed-size pool of GPUParticle structs living in a storage buffer, a compute pass that ran a force-integration kernel (ParticleForce.comp / particleForce) and an Euler integrator (ParticleIntegrate.comp / particleIntegrate), and a billboard vertex shader that rendered whatever lived in the pool.

The problem: no ECS entity could control any of this. The attractor that pulled particles was hardcoded to the camera-forward direction. Spawning new particles meant calling low-level renderer internals directly — which bypassed the whole scene graph. An entity in the world could not say "emit particles from my position."

The goal was to close that gap without dismantling the renderer's GPU pipeline. The computation stays on the GPU; the ECS drives what gets written into the GPU buffer and where forces come from.

Why Not PR #6?

There was a prior attempt at this (unmerged PR #6) that I had to be careful around. That PR introduced a parallel particle entity with its own naming conventions — it did not follow the *Component / *System suffix pattern used everywhere else in the codebase, and it created new files rather than extending components.hpp and systems.hpp. The surface area was also much larger than necessary at the time.

The decision here was to add directly inside the existing components.hpp and systems.hpp, following the established naming and file structure. No new parallel entity files. PR #6's ideas about a ring buffer were sound; the execution approach was what differed.

Iteration 1 — The Bridge Approach (Discarded)

The first instinct was a minimal bridge: iterate emitter entities, read their world position, and call renderer->setParticleAttractors() with a single entry. The renderer already accepted an attractor position — it just wasn't wired to any entity. This would let an emitter entity act as a gravity well for the existing GPU simulation.

This worked in about 30 lines. But it was fundamentally not ECS. The "emitter" component was not actually emitting anything — particles were spawned by whatever initialization code set initial positions in the GPU buffer, not driven by any entity lifecycle. Calling it an emitter was misleading. The attractor coupling also capped at one entity, which made scenes with multiple emitter sources impossible.

I threw it away and started over with a proper ECS design.

Iteration 2 — True ECS: CPU Spawn Logic + Per-Emitter Slots

Design

The core insight is a division of labor:

  • CPU (ECS system): decides when to spawn, where (world position), and how (initial velocity, color, lifetime). Writes initial particle state into GPU memory.
  • GPU (compute shaders): integrates forces each frame. It does not know or care about entities — it just sees an array of particles and applies physics.

Each ParticleEmitterComponent claims a contiguous range of slots from the global GPU particle pool on first use. The pool is a static array; slots are permanent for the lifetime of the emitter (no dynamic reallocation). Within its range, the emitter maintains a ring cursor and overwrites the oldest slot when a new particle spawns — classic ring buffer behaviour.

Slot Claiming

// In ParticleEmitterSystem::update(), first frame for this emitter:
if (emitter._slotBegin == ~0u) {
    emitter._slotCount = emitter.maxParticles;
    emitter._slotBegin = renderer->claimParticleSlots(emitter._slotCount);
    if (emitter._slotBegin == ~0u) continue; // pool exhausted
    emitter._ringCursor = 0;
}

claimParticleSlots() is a high-water-mark allocator: it returns the current m_particleSlotsAllocated, then advances it by count. If the resulting value would exceed MAX_PARTICLES, it returns ~0u. Simple, no fragmentation, no bookkeeping. The trade-off — there is no real releaseParticleSlots() yet — is documented as a known issue below.

Emission Rate and Accumulator

Spawn timing uses an accumulator rather than a per-frame count, which is standard practice for rate-based systems and handles low frame rates correctly:

emitter._accumulator += dt;
float interval = 1.0f / emitter.emissionRate;
while (emitter._accumulator >= interval) {
    emitter._accumulator -= interval;
    // ... spawn one particle
}

At 20 particles/s and 60 fps the accumulator naturally batches at most one particle per frame. At 200/s it batches up to ~3. The spawned particles are collected into a std::vector<GPUParticle> and flushed to the GPU in a single call.

Ring Buffer Write — A Subtle Bug Fixed

The first ring buffer implementation had the cursor in the wrong order:

// BUG: cursor was advanced first, then used as the write target
emitter._ringCursor = (emitter._ringCursor + spawns.size()) % emitter._slotCount;
uint32_t writeSlot = emitter._slotBegin + emitter._ringCursor; // wrong slot

The fixed version records the write position before advancing:

// CORRECT: record position first, then advance
uint32_t writeSlot = emitter._slotBegin + emitter._ringCursor;
renderer->uploadParticles(writeSlot, spawns);
emitter._ringCursor = (emitter._ringCursor + (uint32_t)spawns.size()) % emitter._slotCount;

Cone-Sampled Velocity

Initial velocity is sampled uniformly inside a cone around emitDirection (local-space, rotated by the entity's world transform). The perpendicular basis construction handles the degenerate case where worldDir is nearly aligned with the X axis:

glm::vec3 perp = std::abs(worldDir.x) < 0.9f
    ? glm::vec3(1, 0, 0) : glm::vec3(0, 1, 0);
glm::vec3 u = glm::normalize(glm::cross(worldDir, perp));
glm::vec3 v = glm::cross(worldDir, u);
float theta = unit(rng) * 2.0f * 3.14159265f;
float phi   = unit(rng) * emitter.spread;
glm::vec3 dir = glm::normalize(
    worldDir * std::cos(phi) + (u * std::cos(theta) + v * std::sin(theta)) * std::sin(phi)
);

GPU Struct Alignment — The _pad Repurposing

The existing GPUParticle struct had two mystery padding fields:

struct alignas(16) GPUParticle {
    glm::vec3 position; float _pad1; // after vec3 (12 bytes) → pad to 16
    glm::vec3 velocity; float _pad2;
    glm::vec3 force;    float _pad3;
    glm::vec4 color;
}; // 64 bytes

These pads existed because GLSL std430 (and Metal device address space) require float3 to be 12 bytes but the next field must still land on a 4-byte boundary — and the overall struct must be a multiple of the largest member alignment (16). The pads had always been there and always been zero.

Repurposing _pad1 → lifetime and _pad2 → age is a zero-cost change: same layout, same binary size, same alignment. The CPU initializes them during spawn; the GPU integrator increments age each frame and uses both to compute a fade factor and cull dead particles. _pad3 stays as padding since there is no obvious semantic use for it yet.

struct alignas(16) GPUParticle {
    glm::vec3 position; float lifetime; // seconds; 0 = immortal
    glm::vec3 velocity; float age;      // incremented by GPU integrator
    glm::vec3 force;    float _pad3;
    glm::vec4 color;
};

The Attractor Array Bug — A Nasty Alignment Trap

This one took a moment to spot. The original ParticleAttractorData had a spurious padding field that looked innocuous:

struct alignas(16) ParticleAttractorData {
    glm::vec3 position; // offset 0
    float     _pad1;    // offset 12
    float     strength; // offset 16
}; // 20 bytes on CPU, but GLSL struct was different

The GLSL attractor struct in the compute shader was:

struct AttractorData { vec3 position; float strength; }; // std430: 16 bytes

GLSL std430 packs vec3 as 12 bytes + 4 bytes padding = 16 before the next field. So GLSL's strength was at byte offset 12 within the struct — but the CPU was writing the actual strength float at offset 16 (after _pad1). The GPU was reading offset 12, which was _pad1 = 0.0f. Every attractor had zero strength. The particles moved as if there were no attractor at all.

The fix: remove _pad1. The struct becomes exactly 16 bytes (12 for vec3 + 4 for float), which matches the GLSL layout perfectly. As a bonus, it maps cleanly to a vec4 in GLSL (xyz=position, w=strength), which is how the multi-attractor array is expressed in the final shader to avoid std140 ambiguity entirely:

// GLSL — avoid struct padding confusion, use vec4 directly
layout(set=0, binding=2) readonly buffer AttractorBuffer {
    vec4 attractors[MAX_ATTRACTORS]; // xyz=position, w=strength
};

Multi-Attractor Architecture

Decision

Expand the single hardcoded attractor to an array of up to MAX_PARTICLE_ATTRACTORS = 8 attractors, populated each frame by a dedicated ParticleForceFieldSystem.

Two Sources of Attractors

ParticleForceFieldSystem::update() collects attractors from two sources:

  1. Dedicated attractor entities — any entity with TransformComponent + ParticleAttractorComponent. These exist solely to pull particles toward a point in space, decoupled from any emitter.
  2. Emitter entities — any ParticleEmitterComponent with attractorStrength > 0 also registers itself as an attractor. This lets emitters pull their own particles back toward the spawn point (a common artistic effect).

Separating ParticleForceFieldSystem from ParticleEmitterSystem was a deliberate ECS choice: force fields and emission are orthogonal concerns. An attractor entity has no emission logic; an emitter entity may or may not act as a force field. The split also makes it easy to add new force types (repulsors, vortices) without touching the emitter system.

Capacity

Eight attractors covers the expected artistic use cases (main emitter + a few environmental pulls) without bloating the uniform buffer. The GPU loop is cheap for small N; if we ever need more, MAX_PARTICLE_ATTRACTORS is a single constant.

Wind — The Naming Decision

The first draft of the wind component was called ParticleWindComponent. That name is wrong. Wind is a world-space phenomenon — vegetation sways in it, cloth sims read from it, audio occlusion varies with it, and particles react to it. Naming it after particles ties a world-concept to a single subsystem.

It was renamed to WindFieldComponent and placed in the general components section alongside other world-state concepts. The particle system reads it; other systems can read it independently without any coupling to particle code.

struct WindFieldComponent {
    glm::vec3 direction = {1.0f, 0.0f, 0.0f};
    float     strength  = 0.0f;
    bool      enabled   = true;
};

Singleton-style: attach to one "world settings" entity. First enabled instance wins. If no wind entity exists, strength stays 0 and the GPU wind term is a no-op.

Persistent GPU Buffer Mapping (Vulkan)

The original Vulkan particle buffer was mapped with vkMapMemory / vkUnmapMemory every time a CPU write was needed. That is fine for infrequent updates but wrong for a system that uploads particle data every frame.

The particle buffers were switched to createBufferMapped(), which maps the memory at allocation time and keeps it mapped for the lifetime of the buffer. uploadParticles() becomes a plain memcpy:

void Renderer_Vulkan::uploadParticles(uint32_t slotBegin, const std::vector<GPUParticle>& particles) {
    if (!particleSystemEnabled || particles.empty()) return;
    if (slotBegin + particles.size() > MAX_PARTICLES) return;
    size_t bytes = particles.size() * sizeof(GPUParticle);
    for (size_t i = 0; i < MAX_FRAMES_IN_FLIGHT; i++) {
        auto* dst = static_cast<GPUParticle*>(particleBuffersMapped[i]) + slotBegin;
        memcpy(dst, particles.data(), bytes);
    }
}

Writing to all MAX_FRAMES_IN_FLIGHT (3) buffers on spawn is important. The renderer is triple-buffered: at any moment the GPU is reading frame N-1 while the CPU is preparing frame N. If we only wrote to the current frame's buffer, the next frame's buffer would still contain stale (or zeroed) particle state. Writing to all three on spawn means the initial state is consistent across the entire pipeline regardless of which frame slot is active when the first compute pass runs.

Particle Expression Skeletons

The Ghost of Tsushima talk on particles-as-cinematic-language was the inspiration here. The idea: particles are not just visual effects — they carry emotional meaning. A scene in the "Calm" state has slow-drifting, widely-spaced emitters; "Tense" compresses them, increases rate, shifts color toward red. These are not gameplay mechanics, they are cinematography tools.

The full implementation is deferred, but the scaffolding is in place:

SimulationParams Layout — std140 Verification

Adding attractorCount and wind to ParticleSimulationParams required verifying the struct's std140 layout carefully, since a single wrong padding assumption means the GPU reads garbage.

struct alignas(16) ParticleSimulationParams {
    glm::vec2 resolution;     // offset  0, size 8
    glm::vec2 mousePosition;  // offset  8, size 8
    float     time;           // offset 16, size 4
    float     deltaTime;      // offset 20, size 4
    uint32_t  particleCount;  // offset 24, size 4
    uint32_t  attractorCount; // offset 28, size 4
    glm::vec4 wind;           // offset 32, size 16
}; // total: 48 bytes — multiple of 16 ✓

No padding holes anywhere: two vec2s fill the first 16 bytes, two floats + two uints fill the next 16, and the final vec4 is naturally 16-byte aligned. The GLSL uniform block uses the same layout. This was verified by offset arithmetic, not assumed.

Metal vs. Vulkan Shader Parity

The Metal shaders in 3d_particle.metal and the Vulkan GLSL shaders were rewritten in parallel to maintain feature parity:

  • Both have an attractor loop over up to MAX_ATTRACTORS entries.
  • Both apply inverse-square-law attraction with a softening term to avoid singularities at near-zero distance.
  • Both apply wind as a constant velocity bias scaled by params.wind.w.
  • Both increment age per frame and compute a fade-out in the last 30% of lifetime.
  • Both hide dead particles (age > lifetime) by setting alpha to 0 and moving the billboard off-screen.

One inconsistency found in review: the damping coefficient is 0.1 in Metal and 0.5 in Vulkan. This is a medium-priority bug — particles decelerate at different rates on the two backends. Not tracked as a separate ADR decision since it was an oversight, not a deliberate choice.

Known Issues

CRITICAL Metal uploadParticles() only writes to the current frame's buffer. Vulkan correctly writes to all MAX_FRAMES_IN_FLIGHT buffers. On Metal, particles uploaded in frame N will be invisible (stale data) in frames N+1 and N+2. Fix: iterate over all Metal particle buffers the same way Vulkan does.
MEDIUM Damping coefficient mismatch: Metal uses 0.1, Vulkan uses 0.5. Particles decelerate at different rates across backends. Both values should be unified in a shared constant.
MEDIUM releaseParticleSlots() is a no-op. If an entity with a ParticleEmitterComponent is destroyed at runtime, its GPU slots are never returned to the pool. Long-running sessions that create and destroy many emitters will exhaust MAX_PARTICLES. A free-list allocator would fix this.
LOW The overflow guard in uploadParticles() (slotBegin + particles.size() > MAX_PARTICLES) does not handle the case where slotBegin == ~0u — unsigned arithmetic wraps and the check passes spuriously. Guard should be slotBegin == ~0u || slotBegin + particles.size() > MAX_PARTICLES.

Alternatives Considered and Rejected

OptionWhy Rejected
Pure GPU spawning (spawn compute shader reads from entity buffer) Requires uploading the entire ECS entity state to the GPU each frame. The spawning rules (cone angle, spread, emission rate accumulator) would need to be reimplemented in GLSL. CPU logic is simpler, easier to debug, and fast enough for the particle counts we target.
Per-emitter triple-buffered sub-pools Avoids writing to all frame buffers on spawn, but requires the ring cursor to advance independently per frame. Significantly more complex bookkeeping for a gain that matters only when spawn rates approach the pool size boundary. Deferred.
Single global attractor (camera-forward hardcoded) Original design. Cannot support scenes with multiple emitter sources; artistically limiting.
Attractor as part of ParticleEmitterSystem (no ForceFieldSystem) Conflates emission with force field application. Non-emitter entities could not be attractors. WindFieldComponent would have no natural home.
PR #6 approach (new parallel entity files, non-standard naming) Did not follow established *Component / *System naming, introduced parallel file structure, was larger in scope than needed at the time. Would have been harder to review and merge cleanly.

Call Order

The correct system call order each frame, from earliest to latest:

// 1. Update world transforms
TransformSystem::update(registry);

// 2. Upload force fields to renderer (reads transform positions)
ParticleForceFieldSystem::update(registry, renderer);

// 3. Spawn new particles (reads transform positions, writes GPU buffer)
ParticleEmitterSystem::update(registry, renderer, dt);

// 4. (Future) Modulate emitter parameters from EmotionState
// EmitterModulatorSystem::update(registry, state);

// 5. GPU compute passes run inside renderer->draw():
//    a. ParticleForce.comp / particleForce — apply attractors + wind
//    b. ParticleIntegrate.comp / particleIntegrate — Euler step, age, fade
//    c. Particle.vert + billboard draw call
renderer->draw(registry, scene, camera);

Future Work

  • Implement EmitterModulatorSystem: map EmotionState to per-emitter parameter curves. The scaffolding is in place; the lookup table / curve data structure is the remaining design question.
  • Implement ParticleBurstSystem: sphere-sampled one-shot spawns, tag removal.
  • Fix the three known issues above (Metal multi-frame upload, damping unification, slot release).
  • Consider a free-list allocator for claimParticleSlots / releaseParticleSlots if dynamic emitter create/destroy becomes a real use case.
  • Per-particle texture atlas support: the GPUParticle struct has one spare float (_pad3) that could encode a frame index.
