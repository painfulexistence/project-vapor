1. Context & Motivation

Project Vapor had a self-contained GPU orbital particle demo — 1 000–8 192 particles orbiting a camera-tracking attractor, computed by two Metal/Vulkan compute kernels and rendered as instanced billboards. Beautiful, but completely hardwired.

The question was: can an EnTT entity own a particle emitter and drive it through the normal ECS update loop, while sharing the same GPU buffer the demo already runs on?

Three non-goals from the start:

  • No separate "particle entity" buffer — one shared GPU pool, slot-allocated per emitter.
  • No CPU-side simulation — forces stay on GPU; the ECS only spawns particles and sets per-frame force parameters.
  • No breaking changes — the orbital demo keeps working; emitters coexist alongside it.

2. Architecture Overview

Component layer (components.hpp)
Pure data, no logic, no renderer references.
  • ParticleEmitterComponent — spawn rate, speed, spread, direction, color, and hidden runtime state (_slotBegin, _ringCursor, _accumulator).
  • ParticleAttractorComponent — strength scalar; position comes from the co-resident TransformComponent.
  • WindFieldComponent — world-space wind (direction + strength). Named "WindField", not "ParticleWind", because vegetation, cloth, and audio also care about wind.
  • ParticleBurstRequest — one-shot component: system processes it, then removes it from the entity.
  • EmitterModulatorComponent — skeleton for emotion-driven particle expression (GoT-style).
  • SpellBoltComponent — origin/target/speed/arcHeight; drives a moving emitter along a Bezier arc.
System layer (systems.hpp)
Six static-method systems; each takes the registry + renderer (IRenderer*) + optional deltaTime.

3. GPU Data Layout

GPUParticleData (64 bytes)

The two "pad" fields that std430/MSL alignment forced on vec3 were repurposed. Zero change in binary layout.

struct alignas(16) GPUParticleData {
    glm::vec3 position;  float lifetime; // -1 = immortal
    glm::vec3 velocity;  float age;      // seconds alive
    glm::vec3 force;     float _pad3;
    glm::vec4 color;
};

ParticleSimParams (extended to 64 bytes)

Added attractorCount, wind, and turbulence. The uint that was _pad2 became attractorCount — zero offset shift.

struct alignas(16) ParticleSimParams {
    glm::vec2 resolution;    glm::vec2 mousePosition; // offset 0
    float     time;          float deltaTime;         // offset 16
    uint32_t  particleCount; uint32_t attractorCount; // offset 24
    glm::vec4 wind;       // xyz=dir, w=strength      // offset 32
    glm::vec4 turbulence; // w=strength               // offset 48
}; // 64 bytes

ParticleAttractor (32 bytes)

Layout: vec3 position (12) + _pad0 (4) + float strength (4) + _pad1[3] (12) = 32 bytes. MSL float3 aligns to 16 bytes, so strength lands at offset 16 in both C++ and shaders.

4. Particle Slot Management

The GPU particle buffer is a flat array of MAX_PARTICLES entries. Emitters carve out contiguous sub-ranges using a first-fit free list with adjacent-range coalescing on release.

struct ParticleSlotRange { uint32_t begin = 0, count = 0; };
std::vector<ParticleSlotRange> m_particleSlotFreeList;

// Allocate: first range with enough capacity, split remainder
// Release:  push range, sort by begin, merge adjacent neighbors

Within a claimed range, each emitter uses a ring cursor. Spawn n particles per frame → overwrite the next n slots, wrapping around. The per-particle lifetime/age fields ensure dead particles fade out naturally without needing to track them CPU-side.

Why ring buffer? The alternative — free-list inside the emitter's own sub-range — requires tracking individual live particles on the CPU. Ring buffer is O(1) and lets the GPU kill particles by age without CPU coordination.

5. Force Field & Wind System

The attractor buffer grew from a single ParticleAttractor struct to an array of MAX_PARTICLE_ATTRACTORS = 8. The force shader loops over attractorCount active attractors using an inverse-square law with a softening term of 0.1:

for (uint i = 0; i < attractorCount; i++) {
    vec3  dir      = normalize(attractors[i].position - p.position);
    float forceMag = attractors[i].strength / (dist * dist + 0.1);
    p.force += dir * forceMag;
}

Wind is additive: p.force += wind.xyz * wind.w. The WindFieldComponent is intentionally not called ParticleWindComponent — it's a world-space concept that other systems (foliage, cloth, ambient sound) should also read.

6. Divergence-Free Curl Noise

Simple gradient noise creates visible compression artifacts in particle trajectories — particles cluster around noise extrema. True curl noise takes the curl (∇×) of a 3-component potential field, which is by construction divergence-free: no sources, no sinks, particles flow smoothly.

The implementation uses 12 finite-difference evaluations across 3 decorrelated vnoise potential fields:

vec3 curlNoise(vec3 p, float t) {
    const float e = 0.1;
    vec3 q = p * 0.5 + vec3(t * 0.07);

    float dFzdy = vnoise(q+o1+vec3(0,e,0)) - vnoise(q+o1-vec3(0,e,0));
    float dFydz = vnoise(q+o2+vec3(0,0,e)) - vnoise(q+o2-vec3(0,0,e));
    // ... 4 more pairs ...
    return vec3(dFzdy-dFydz, dFxdz-dFzdx, dFydx-dFxdy) / (2.0*e);
}

At 8 192 particles per frame this adds ~98K noise evaluations to the force pass — negligible on modern GPUs. Turbulence strength is tunable per-emitter via IRenderer::setParticleTurbulence().

7. The Spell Bolt

The design brief was "Hogwarts Legacy wand-shot feel: known origin → target, particles along the trajectory, but with a fluid quality."

Two approaches considered:

  • Spline force field — attract existing pool particles toward a Bezier curve. Gives "river along a path" aesthetics. Not right here — the spell bolt is a moving emitter, not a field.
  • Moving emitter + curl noise — a ParticleEmitterComponent rides a Bezier arc, emitting along the tangent direction; curl noise adds fluid swirl to individual particle trajectories. Chosen.

SpellBoltSystem drives the quadratic Bezier:

glm::vec3 mid = (origin + target)*0.5f + vec3(0, arcHeight, 0);
float s = clamp(progress, 0, 1);
t.position = (1-s)*(1-s)*origin + 2*(1-s)*s*mid + s*s*target;

// Tangent → emitter direction
glm::vec3 tan = normalize(2*(1-s)*(mid-origin) + 2*s*(target-mid));
emit.emitDirection = tan;

On arrival (progress ≥ 1.0), the system disables the emitter, adds a ParticleBurstRequest for the impact explosion, and removes the SpellBoltComponent.

8. Bugs & Fixes

Ring buffer cursor advance

The initial implementation called uploadParticles(emit._slotBegin + emit._ringCursor, batch) after advancing _ringCursor. Off-by-one: particles wrote one position ahead of where they should.

Fix: capture writeSlot = _slotBegin + _ringCursor before the modular advance.

ParticleAttractor alignment trap (GLSL)

An earlier attempt placed an explicit _pad1 between position and strength in the C++ struct, but the matching GLSL struct had implicit padding (vec3 → 16 bytes). The GLSL attractorStrength was silently reading the pad bytes (always 0), so attractors had zero effect.

Fix: the C++ ParticleAttractor uses _pad0 after position (matching GLSL's implicit vec3 pad) and _pad1[3] after strength to reach 32 bytes total. Both layouts agree at byte level.

Damping mismatch between backends

The Metal shader used p.force -= p.velocity * 0.1 while the Vulkan shader used 0.5. Particles spiraled differently per backend.

Fix: both shaders now use 0.5. Softening term also unified to 0.1.

Slot overflow guard

uploadParticles() checked slotBegin + particles.size() > MAX_PARTICLES, but when slotBegin == ~0u the addition wraps to a valid range on 32-bit unsigned arithmetic.

Fix: guard with if (slotBegin == ~0u || …) return; first.

Slot free-list was a no-op

releaseParticleSlots() did nothing in the initial implementation — particles were claimed but never returned to the pool.

Fix: full first-fit allocator with coalescing on release in both Renderer and Renderer_Metal.

Metal used wrong struct types

The Metal renderer was initialized with GPUParticle (from graphics_effects.hpp) and local ParticleSimParams/ParticleAttractor structs, diverging from the canonical types in render_data.hpp.

Fix: Metal renderer now uses GPUParticleData, ParticleSimParams, and ParticleAttractor from render_data.hpp throughout. The particle buffer init and the ParticlePass local structs removed.

9. Metal vs. Vulkan Divergences

ConcernVulkanMetal
Particle buffer memory CPUtoGPU / HOST_VISIBLE via RHI updateBuffer() StorageModeShared — CPU writes directly to GPU-readable memory
Triple-buffer uploads Sim-params + attractor buffers are frame-slotted (createFrameSlottedBuffer) Sim-params + attractor buffers are triple-buffered; particle data buffer is single (shared)
Shader binding order params(set0,b0), attractors(set0,b1), particles(set0,b2) particles(b0), params(b1), attractors(b2)
Compute barrier Explicit rhi->computeBarrier() between force and integrate passes Two separate setComputePipelineState calls inside one encoder (Metal orders them automatically)
Why Metal doesn't need multi-frame particle uploads: The particle data buffer uses StorageModeShared — CPU and GPU share the same physical memory. A CPU write is immediately visible to the GPU without staging or copying. Contrast with Vulkan's HOST_VISIBLE model where each frame-in-flight slot must be written separately to avoid race conditions with in-flight GPU reads.

10. Particle Expression (GoT-style)

Ghost of Tsushima used particles for cinematic emotional beats — wind through grass, embers during combat tension, snowflakes in quiet moments. The EmitterModulatorComponent is a skeleton for this:

enum class EmotionState { Neutral, Joy, Rage, Sorrow, Fear };

struct EmitterModulatorComponent {
    EmotionState state = EmotionState::Neutral;
    float transitionSpeed = 1.0f;
    float _blendTimer = 0.0f;
};

To build out: EmitterModulatorSystem reads the current state and lerps emitter parameters (rate, speed, spread, color, turbulence) toward target values. WindFieldComponent strength modulation ties weather to combat intensity. All of this is ECS-native — the renderer just sees float parameters, not game state.

12. Lessons Learned

Naming matters more than you think

The first iteration of the wind component was ParticleWindComponent. Renaming it to WindFieldComponent immediately clarified that wind is a world property, not a particle property — and opened the door for foliage and audio to read the same component without coupling to the particle system.

Pad fields are free real estate

GPU-facing structs are littered with alignment pads. Repurposing _pad1/_pad2 in GPUParticleData as lifetime/age added per-particle lifetime tracking with zero binary-layout change — no shader re-binding, no buffer resize.

Divergence-free curl noise is worth the 12 evaluations

Simple gradient noise creates "sticky" regions where particles cluster. Curl noise (curl of a potential field) guarantees incompressibility — particles flow without accumulating. On a GPU doing 8 192 invocations per frame, 12 extra noise lookups are completely free.

Ring buffer + GPU lifetime ≫ CPU particle tracking

A traditional particle system maintains a CPU-side list of live particles. The ring-buffer approach skips that entirely: overwrite old slots, let age/lifetime handle fade-out on the GPU. The CPU only cares about where to write, not who is alive.

One pool, many emitters

Sharing a single GPU buffer across all emitters via a first-fit free list means zero buffer synchronization between emitters. The GPU sees one big array; the ECS carves it up logically. The downside: a large emitter claiming most of MAX_PARTICLES squeezes everyone else. The fix is raising MAX_PARTICLES (currently 8 192) — a one-line constant change.
