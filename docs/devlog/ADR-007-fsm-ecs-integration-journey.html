ADR-007: The FSM + ECS Integration Journey

2026-05-30 Adopted

This document chronicles the iterative process of building a Playmaker-style Finite State Machine system that actually fits ECS principles. It took three major rewrites to get right. Here's what we learned.

The Problem: FSM and ECS Don't Mix Naturally

Traditional FSM implementations are inherently object-oriented. A state machine typically looks like this:

class PlayerStateMachine {
    State* currentState;
    void onEnter() { currentState->enter(); }
    void onExit() { currentState->exit(); }
    void update() { currentState->update(); }
};

This is fundamentally at odds with ECS, where:

  • Components are pure data — no methods, no callbacks
  • Systems are stateless functions that iterate over components
  • Behavior emerges from the composition of systems, not inheritance

The challenge: how do you represent state transitions, enter/exit callbacks, and conditional logic without polluting your components with behavior?

Attempt #1: The OOP Trap (Rejected)

The first implementation looked reasonable at first glance:

struct FSMComponent {
    uint32_t currentState;
    std::function<void()> onEnter;  // Callback
    std::function<void()> onExit;   // Callback
    std::vector<Transition> transitions;
};
Pitfall #1: Callbacks break serialization.
You can't serialize std::function. This makes save/load impossible and debugging a nightmare — you can't inspect what a callback does from the component data alone.
Pitfall #2: Callbacks create hidden dependencies.
A callback might capture a pointer to another system, creating coupling that's invisible to the ECS. When that system is destroyed, you get dangling references with no compile-time safety net.

This approach was scrapped within hours. Back to the drawing board.

Attempt #2: Event-Driven, But Wrong Abstraction (Rejected)

The second attempt moved callbacks out of components into "Request" components:

// Engine layer
struct FSMStateChangeRequest {
    uint32_t targetState;
    std::function<void()> onComplete;  // Still a callback!
};
Pitfall #3: Request components in the engine layer are too opinionated.
The engine shouldn't know what "requests" the game wants to make. ShowSubtitleRequest is game logic, not engine infrastructure.

We were conflating two concerns:

  1. FSM mechanics (state, transitions, timers) — belongs in engine
  2. What to do on state change (show UI, play sound) — belongs in game

Attempt #3: Pure Data + Event Components (Adopted)

The breakthrough came from asking: what if state changes are just another component?

// Engine emits this when a transition occurs
struct FSMStateChangeEvent {
    uint32_t fromState;
    uint32_t toState;
    float previousStateTime;
};

// Game systems react to it
class SubtitleActionSystem {
    static void update(entt::registry& reg) {
        auto view = reg.view<SubtitleQueueComponent, FSMStateChangeEvent>();
        for (auto entity : view) {
            auto& event = view.get<FSMStateChangeEvent>(entity);
            if (event.toState == SubtitleStates::WaitingForVisible) {
                // Trigger the action here — game logic, not engine
                PageSystem::show(reg, PageID::Subtitle);
            }
        }
    }
};
Key insight: The FSM system's only job is to manage state and emit FSMStateChangeEvent. What happens in response is entirely the game's responsibility.

The Final Architecture

Data Flow

Sensor Systems → FSMEventQueue → FSMSystem → FSMStateChangeEvent → Action Systems
     (input)        (pending)      (process)      (output)           (react)

Components (Engine Layer)

FSMDefinition      // States, transitions, timed transitions
FSMStateComponent  // Current state, timers
FSMEventQueue      // Pending events to process
FSMStateChangeEvent // Output: "state just changed from X to Y"

Systems

FSMInitSystem::update()  // Auto-initialize entities with FSMDefinition
FSMSystem::update()      // Process events, check timers, emit FSMStateChangeEvent
TriggerSystem::update()  // Convert physics triggers to ECS events

Pitfalls We Hit Along the Way

Pitfall #4: Systems with multiple static methods.
We initially had FSMSystem::init() and FSMSystem::update(). This broke the "one system, one update" principle from Bevy. Solution: split into FSMInitSystem (reactive initialization) and FSMSystem (per-frame update).
Pitfall #5: Helper classes that hide component manipulation.
FSMEventHelper::send() looked convenient but was just wrapping reg.get_or_emplace<FSMEventQueue>(e).push(event). We deleted it — callers should see the component manipulation explicitly.
Pitfall #6: Timed transitions checked before timer update.
The FSM system checks stateTime >= duration then updates stateTime += dt. This means a timed transition fires on the next frame after the duration is reached, not the same frame. Our tests initially got this wrong.
Pitfall #7: TriggerVolumeComponent had callbacks.
The existing trigger system used std::function<void(Entity)> onEnter. We replaced it with TriggerEnterEvent / TriggerExitEvent components that game systems can query.

How to Use It

1. Define Your FSM

namespace PlayerStates {
    constexpr uint32_t Idle = 0;
    constexpr uint32_t Walking = 1;
    constexpr uint32_t Jumping = 2;
}

Vapor::FSMDefinition createPlayerFSM() {
    return Vapor::FSMDefinitionBuilder()
        .state("Idle")
        .state("Walking")
        .state("Jumping")
        .transition("Idle", "Walking", "StartWalk")
        .transition("Walking", "Idle", "Stop")
        .transition("Idle", "Jumping", "Jump")
        .timedTransition("Jumping", "Idle", 0.5f)  // Auto-land after 0.5s
        .initialState("Idle")
        .build();
}

2. Attach Components

auto player = registry.create();
registry.emplace<Vapor::FSMDefinition>(player, createPlayerFSM());
registry.emplace<Vapor::FSMEventQueue>(player);
// FSMInitSystem will auto-add FSMStateComponent on next frame

3. Send Events

// From input system or game logic:
registry.get<Vapor::FSMEventQueue>(player).push("StartWalk");

4. React to State Changes

class PlayerAnimationSystem {
    static void update(entt::registry& reg) {
        auto view = reg.view<PlayerComponent, Vapor::FSMStateChangeEvent>();
        for (auto [entity, player, event] : view.each()) {
            if (event.toState == PlayerStates::Walking) {
                player.animator.play("walk");
            } else if (event.toState == PlayerStates::Jumping) {
                player.animator.play("jump");
                // Spawn dust particles, play sound, etc.
            }
        }
    }
};

Extending: Hierarchical FSM

To support nested state machines (e.g., Combat.Attacking.Slash), extend the components:

struct HierarchicalFSMComponent {
    std::vector<uint32_t> stateStack;  // [Combat, Attacking, Slash]

    uint32_t currentLeafState() const { return stateStack.back(); }
    bool isInState(uint32_t state) const {
        return std::find(stateStack.begin(), stateStack.end(), state) != stateStack.end();
    }
};

// Transitions can now target parent states
struct HierarchicalTransition {
    std::vector<uint32_t> fromPath;  // Match prefix
    std::vector<uint32_t> toPath;    // New state path
    std::string event;
};

The system remains the same — it just operates on paths instead of single state IDs.

Extending: Actions on Transition

For "fire-and-forget" actions (particles, sounds), use action request components:

// Define action request components
struct PlaySoundRequest { std::string soundId; };
struct SpawnParticlesRequest { std::string effectId; glm::vec3 position; };

// Action system reacts to state changes and emits requests
class PlayerEffectsSystem {
    static void update(entt::registry& reg) {
        auto view = reg.view<Vapor::FSMStateChangeEvent, TransformComponent>();
        for (auto [entity, event, transform] : view.each()) {
            if (event.toState == PlayerStates::Jumping) {
                reg.emplace<PlaySoundRequest>(entity, PlaySoundRequest{"jump"});
                reg.emplace<SpawnParticlesRequest>(entity,
                    SpawnParticlesRequest{"dust", transform.position});
            }
        }
    }
};

// Separate systems consume the requests
class SoundSystem {
    static void update(entt::registry& reg, AudioEngine* audio) {
        auto view = reg.view<PlaySoundRequest>();
        for (auto [entity, request] : view.each()) {
            audio->play(request.soundId);
        }
        reg.clear<PlaySoundRequest>();  // Consume all requests
    }
};
Pattern: State change → Effect system emits Request components → Dedicated systems consume requests. This keeps effects decoupled from state logic.

Lessons Learned

  1. Callbacks are poison in ECS. If you need behavior on state change, emit an event component. Let another system handle it.
  2. Keep engine and game concerns separate. The engine provides FSM mechanics. The game decides what those states mean.
  3. One system, one update. If you have init() and update(), split them into two systems.
  4. Helper classes hide the truth. Let callers see the component manipulation. It's not that much more code, and it's infinitely more debuggable.
  5. Test the timing. FSM timing is subtle. Write tests that check exactly when transitions fire relative to timer updates.
"The best FSM is one where you can look at the components and understand the entire state of the system without reading any code."
