ADR-010: Separating a ParticleRendererComponent

2026-07-17 Proposed

Context

The ECS particle system (ADR — particle integration) split emission, force fields, and slot management into components and systems, but rendering configuration was left entirely in the shaders. A ParticleEmitterComponent decides where, when, how fast, and what colour particles are born; nothing decides how they look beyond that single colour. This ADR records the decision to extract a ParticleRendererComponent — the appearance module, analogous to Unity's separation of the Emission module from the Renderer module — and, more importantly, the architectural constraint that dictates how far that separation can go.

Where appearance currently lives

Particle rendering is 100% GPU. The CPU contributes exactly three inputs per frame: the particle SSBO contents, the camera, and a single global particleSize push constant (hardcoded 0.1 in both backends). The rest is fixed in two shaders:

  • Vertex (Particle.vert / particleVertex): reads the particle, builds a camera-facing billboard quad of size particleSize, forwards color.
  • Fragment (Particle.frag / particleFragment): a soft radial disc plus an additive glow term, discards near-transparent pixels.

So the only per-particle appearance channel the CPU feeds is color, set at emission from emitter.color. Size, shape, glow, blend, and the aging fade are all global constants baked into the shader.

The current aging model, written out

A recurring question — "as a particle ages, does it shrink?" — is worth pinning down precisely, because it is the baseline this ADR proposes to make configurable. The radius is constant; only opacity fades. The fade lives in the integrate compute kernel. With normalized age t = age / lifetime ∈ [0,1] (immortal particles, lifetime < 0, never fade):

α_life(t) = 1 for t ≤ 0.7 = (1 − t) / 0.3 for 0.7 < t < 1 (linear 1 → 0 over the last 30%) = dead, not drawn for t ≥ 1

The fragment shader then applies a fixed radial profile inside the constant-size sprite. With r = 2·|uv − 0.5| ∈ [0,1] (0 = centre, 1 = edge):

opacity(t, r) = α_life(t) · (1 − smoothstep(0, 1, r)) rgb(r) = color · (disc(r) + 0.5 · e^(−2r²)) disc(r) = 1 − smoothstep(0,1,r)

There is no size term of t anywhere. A "shrink to a point on death" look — a size-over-life curve — does not exist today; it is exactly the kind of knob a renderer component should own.

The constraint that shapes the decision

Rendering is a single instanced draw over [0, particleCount) with one pipeline and one push-constant block. The draw has no idea which particle belongs to which emitter — emitters own contiguous slot ranges, but that ownership lives on the CPU and is never handed to the GPU draw. Therefore a ParticleRendererComponent cannot simply be "read at draw time." Per-emitter appearance requires one of three mechanisms, and they differ in what they can express:

Option A — bake scalar params per particle

Widen the per-particle appearance channel. GPUParticleData is 64 bytes with a currently-unused _pad3 float at offset 44 — free space for a per-particle size. The emitter copies its ParticleRendererComponent's parameters into each spawned particle (exactly as color is copied today); the vertex shader reads per-particle size instead of the global push constant. Size-over-life and the fade start can be evaluated in the integrate kernel from t.

  • Enables: per-emitter size, size-over-life, fade curve, colour-over-life — anything expressible as per-particle scalar/vector data.
  • Keeps: the single instanced draw; no CPU-side draw restructuring.
  • Cannot express: pipeline-level state — blend mode, texture/sprite, custom shader — because those are one-per-pipeline, not per-instance.
  • Cost: the free 4-byte slot covers one float (size). More per-particle params mean growing the struct past 64 bytes (× up to 3M particles = real bandwidth), or packing.

Option B — per-emitter draw calls

Draw each emitter's slot range separately, each with its own push constants / pipeline. This is the only way to get per-emitter texture, blend mode, and shader.

  • Enables: full per-emitter appearance including pipeline state.
  • Cost: N draw calls (N = live emitters); the draw must target exactly an emitter's live particles, which interacts awkwardly with the ring buffer (live particles need not be contiguous within the range) and with reclaimed/zeroed holes. State sorting to minimise pipeline switches becomes the renderer's job.

Option C — global render config only

Do not go per-emitter yet. Lift the hardcoded constants (size, glow intensity, fade-start fraction, soft-edge) into one tweakable settings block — a global component or a renderer struct surfaced in the Systems > Particles panel.

  • Enables: live-tunable global look; removes magic numbers from shaders.
  • Cannot express: anything per-emitter — both demo emitters would share one size/glow/fade.
  • Cost: minimal; a uniform block and a few sliders.

Decision

Adopt Option A as the shape of ParticleRendererComponent, with Option C's global block folded in as the defaults, and Option B deferred until a concrete need for per-emitter textures or blend modes appears.

Rationale: A is the smallest change that delivers genuinely per-emitter appearance (the point of separating the component at all) without restructuring the single-draw pipeline. It reuses the existing "copy emitter data into each particle at spawn" path that already carries color, and it fills the free _pad3 slot with the one parameter that most wants to vary per particle — size. Pipeline-state parameters (texture, blend) are explicitly out of scope for this component version; they are a pipeline concern, not a per-particle one, and forcing them into A would mislead.

Proposed component shape

struct ParticleRendererComponent { float size = 0.1f; // world-space billboard half-extent float sizeEndScale = 1.0f; // size multiplier at end of life (1 = constant; <1 shrinks) float fadeStart = 0.7f; // normalized age at which alpha starts ramping to 0 float glow = 0.5f; // additive glow weight in the fragment shader glm::vec4 colorEnd = ...; // optional colour-over-life target (or reuse emitter.color) // blend mode / texture intentionally absent — see Option B, deferred. };

Ownership follows the same rule the emitter fields settled on: gameplay writes these, the system reads them and bakes the per-particle channels (size, and later colour-over-life) into GPUParticleData at spawn. Size-over-life and fade-start are evaluated GPU-side in the integrate kernel from t. An emitter with no ParticleRendererComponent falls back to the global defaults, so the component is purely additive.

Consequences

  • The aging model gains a size term: size(t) = size · lerp(1, sizeEndScale, t), and fadeStart replaces the hardcoded 0.7 in α_life. "Shrink to a point on death" becomes sizeEndScale = 0.
  • GPUParticleData's _pad3 becomes size; the 64-byte layout and Metal float4 aliasing are unchanged (the slot already exists at offset 44).
  • The global particleSize push constant is replaced by per-particle size; the push-constant block can carry the remaining global defaults or be dropped.
  • Per-emitter texture and blend mode remain unavailable until Option B (per-emitter draws) or a texture-atlas + per-particle texture-index scheme is added. This is a deliberate, documented limitation, not an oversight.
  • None of this addresses the CPU-free-list / high-water-mark scaling limits (tracked separately as the GPU-driven-particles proposal); it is orthogonal — appearance vs allocation.

Status: Proposed. Supersedes the hardcoded global particleSize and the fixed last-30% fade. Blocked on nothing; scoped to Option A for the first implementation.
