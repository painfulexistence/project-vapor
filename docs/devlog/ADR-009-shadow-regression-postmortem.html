ADR-009: Shadow Regression Postmortem & Verification Discipline

2026-07-03 Adopted

What happened

A batch of rendering changes landed together on one branch: per-instance normal-matrix precompute, material-ID varyings, TLAS rebuild skipping, any-hit shadow rays, a temporal + à-trous AO denoise chain at half resolution, and shadow texture format/mip changes. Some time later, ray-traced shadows broke: coverage shrank, unexplained bright bands appeared, and the shadow map content looked compressed.

The debugging that followed consumed far more time than the features had. Suspects were reverted one at a time — shadow texture format, mip chain, TLAS refit — with no effect, because the artifact had multiple contributing causes stacked on top of each other, and every revert tested only one variable against a background of five other changed ones.

What was actually wrong

Real bugs found along the way

  • Unguarded CPU normal-matrix inverse. The in-shader inverse() helper returns identity when |det| < 1e-6; the CPU precompute that replaced it called glm::inverse unguarded. Any instance with a degenerate model matrix (a cube flattened into a floor by a zero scale axis) produced inf/NaN normals, which poisoned the normal RT and made shadow-ray origins NaN — whole instances lost their shadows. Lesson: when moving a computation across the CPU/GPU boundary, move its guards with it.
  • Uninitialized MTLAccelerationStructureInstanceDescriptor fields. Only transform, index, and mask were assigned; options and intersectionFunctionTableOffset carried stack garbage since the RT code was first written. Garbage option bits (NonOpaque, winding flags) can make the intersector misclassify whole instances — nondeterministically, since stack contents shift between builds. This is why the artifact appeared "suddenly" after an unrelated rebuild. Lesson: zero-initialize every GPU-facing struct; a POD stack local is not zeroed.
  • The AO pipeline had never been visible. The post-process shader sampled texAO and never used the value — dead code from an old experiment. Every AO change was invisible until the sample was wired to the IBL/ambient term.

Falsely accused

  • shadowRT's R8 format and mip-chain removal (all consumers read .r at LOD 0 — reverted during bisection, exonerated)
  • 1×1-threadgroup dispatch fixes (semantically identical thread grids)
  • Shadow coverage "shrinking" under a near-vertical sun — physically correct behavior misread as a bug while other things were genuinely broken

Resolution

The branch was reset to visual parity with mainline: every rendering-behavior change was backed out in one commit, keeping only tooling, docs, and provably-safe fixes. Shadows immediately verified correct. Features were then reintroduced one at a time, each verified visually before the next: any-hit rays, the AO chain (full res), AO application to the ambient term, half-res shadows. Every step survived.

Discipline adopted

  • One visual change per verification point. Multiple rendering-path changes may share a branch, but never share an unverified interval. The bisect space between checkpoints must stay size one.
  • When stacked changes break, reset to baseline — don't bisect in place. Reverting suspects one at a time against a still-changed background tests nothing. Going back to a known-good state and re-adding is faster than it feels.
  • Make effects observable before iterating on them. The AO work was tuned and re-tuned while its output was disconnected. First verify the wire reaches the screen (the runtime toggle now makes this a one-click A/B).
  • Debug views are prerequisites, not conveniences. The aspect-correct RT previews and per-pass timing rows turned "the shadow looks weird" into "the shadow map's content is wrong, its inputs are right" — the single most useful narrowing step of the hunt.
  • Physically-correct results can look like bugs. Before debugging a visual, check whether the light/camera setup makes the observation expected (vertical sun ⇒ small shadows).

Consequences

  • The merged branch contains tooling, the AO pipeline (SSAO raygen + temporal + à-trous, applied to ambient only), any-hit half-res shadows, and the bug fixes above — each individually verified.
  • Backed-out perf items (normal-matrix precompute with its guard, material-ID varyings, TLAS rebuild skipping, half-res AO) return as separate single-topic PRs.
  • RT soft shadows + RT AO raygen come next on one branch with three verification checkpoints: denoiser generalization (no visual change), cone-sampled shadows, then the AO raygen swap (see ADR-008's plan).
