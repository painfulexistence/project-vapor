ADR-008: Ray Tracing Optimization Notes

2026-07-01 Adopted

Context

The new per-pass GPU timing panel (PR #56) made per-encoder costs visible for the first time, and two numbers stood out immediately: RaytraceShadowPass at ~7.4 ms and RaytraceAOPass at ~6.6 ms — together nearly a full 60 fps frame budget for two secondary effects.

Investigating those numbers uncovered a dispatch mistake shared by four compute passes, and prompted a broader writeup: what actually costs time in GPU ray tracing, which mistakes silently destroy performance, and what a production-shaped RT effect pipeline (with denoising) looks like. The last section uses RT AO as a concrete implementation plan, since the pass currently named RaytraceAOPass is in fact a screen-space AO kernel that never casts a ray.

Background: how compute-based ray tracing executes on Apple Silicon

Metal has no separate ray tracing pipeline stage (no raygen/miss/hit shader tables like DXR or Vulkan RT). Ray tracing is done from ordinary compute (or fragment) kernels via the intersector<> API against acceleration structures:

  • BLAS (primitive acceleration structure) — one per unique geometry; built once, rebuilt only when geometry changes.
  • TLAS (instance acceleration structure) — references BLASs with per-instance transforms; cheap to rebuild or refit per frame when only transforms change.
  • Intersection — on M3 and later, ray/box and ray/triangle tests run on dedicated hardware; on M1/M2 the same API runs a software traversal on the shader cores. Either way the query is issued from a plain compute thread.

Because the entry point is a normal compute kernel, every rule of compute shader performance applies to ray tracing kernels too — and that is exactly where things went wrong.

SIMD execution in one paragraph

Apple GPUs execute threads in SIMD-groups of 32. A threadgroup is carved into SIMD-groups; a SIMD-group always occupies all 32 lanes of hardware whether or not there are 32 live threads in it. Occupancy (how many SIMD-groups are resident per shader core) is what hides memory and intersection latency — RT kernels are latency-heavy, so they depend on it more than most.

The mistake: 1×1 threadgroups

All four screen-space compute passes dispatched like this:

// one threadgroup per pixel, ONE THREAD per threadgroup
encoder->dispatchThreadgroups(MTL::Size(screenSize.x, screenSize.y, 1),
                              MTL::Size(1, 1, 1));

A threadgroup with a single thread still occupies an entire 32-lane SIMD-group: 31 of 32 lanes idle, 1/32 of theoretical instruction-issue throughput, plus per-threadgroup scheduling overhead multiplied by two million (one threadgroup per pixel at full res). Nothing renders incorrectly and no validation layer complains — the kernel just issues ALU work up to 32× slower than it could. How much that costs in wall-clock time depends entirely on whether the kernel is ALU-bound; see the measured outcome below.

The fix is one line per call site. Since all of these kernels index by [[thread_position_in_grid]], dispatchThreads keeps the semantics (one thread per pixel) while packing threads into real threadgroups. Apple-family GPUs support non-uniform threadgroup sizes, so no edge-tile bounds handling is needed on the host side:

// one thread per pixel, 64 threads (two SIMD-groups) per threadgroup
encoder->dispatchThreads(MTL::Size(drawableSize.width, drawableSize.height, 1),
                         MTL::Size(8, 8, 1));

Fixed call sites:

PassGridNote
RaytraceShadowPassfull-resshader unchanged
RaytraceAOPassfull-resshader unchanged
NormalResolvePassfull-resshader unchanged
TileCullingPasstile gridshader index attribute changed from threadgroup_position_in_grid to thread_position_in_grid — its tile coordinate was previously the threadgroup id, which only worked because groups were 1×1

An 8×8 (64-thread) group is a good default for RT and other register-heavy kernels; 16×16 can be better for light kernels, but larger groups increase register pressure and can reduce occupancy. Measure with the timing panel before tuning further.

Measured outcome — and what it actually teaches

On M3, the RT shadow pass time did not measurably change after the fix. The prediction of a severalfold speedup was wrong, and the reason is instructive:

SIMD lane packing buys instruction-issue throughput. The shadow kernel has almost no instructions to issue — a dozen ALU ops to reconstruct the world position, then a single intersect() that executes on the M3's dedicated ray tracing hardware, then one texture write. The pass is bound by ray traversal throughput and BVH memory traffic, neither of which cares how threads were packed into SIMD-groups. Latency hiding comes from the number of resident SIMD-groups, and even 1-thread groups provided enough of them to keep the RT unit fed. Apple's per-threadgroup launch overhead is evidently also cheap enough not to dominate, even at two million threadgroups per dispatch.

The general lesson: classify the kernel before predicting the win. Lane utilization is a multiplier on ALU-bound work only. Where 1×1 threadgroups genuinely hurt:

  • ALU-heavy kernels — long per-thread loops of arithmetic (the 32-sample SSAO loop is a candidate; filtering and denoising kernels are too).
  • M1/M2, where intersect() compiles to a software traversal loop inside the kernel — there the traversal itself is ALU work and lane occupancy multiplies straight through.
  • Kernels using SIMD-scope operations (simd_sum, simd_shuffle) or threadgroup memory, which are meaningless at group size 1.

The fix stays — it is free, correct, and portable — but it is not where the shadow pass's milliseconds live. Reading the kernel with that in mind, the real levers are:

  • Closest-hit vs any-hit: the kernel uses the default closest-hit semantics but only checks hit/none. accept_any_intersection(true) lets traversal stop at the first occluder — the standard shadow-ray optimization.
  • Payload: intersector<instancing, triangle_data> requests per-triangle data (barycentrics) that is never read; plain intersector<instancing> suffices.
  • Ray extent: max_distance = 10000 traverses the whole scene for every miss.
  • Resolution and sample reuse: the half-res + temporal machinery described below applies to shadows as much as to AO.

Checklist: mistakes that silently eat RT performance

  • 1×1 threadgroups — see above. Grep for MTL::Size(1, 1, 1) as the second argument of any dispatch.
  • Rebuilding the TLAS from scratch every frame — TLASBuildPass currently does this (it carries a TODO). At ~0.2 ms for the current scene it is tolerable, but cost scales with instance count. The fix has two tiers: skip the build entirely when no transform changed, and use refit (refitAccelerationStructure) instead of a full rebuild when only transforms changed. Full rebuilds are only needed on instance add/remove.
  • Tracing at full resolution by default — soft, low-frequency effects (AO, diffuse GI, even soft shadows) can be traced at half resolution and upsampled with an edge-aware filter. Half res = 4× fewer rays.
  • Too many rays per pixel instead of reusing rays across time — 32 samples per pixel per frame (what the current AO kernel does in screen space) is the brute-force answer. The production answer is 1–2 samples per pixel plus temporal accumulation: after ~16 frames of history you have effectively 16–32 spp at 1/16 the per-frame cost.
  • White-noise sampling — random(index + frame * 19) style hashes produce high-frequency noise that denoisers struggle with. Blue-noise textures (tiled 128×128, animated by frame index) give error that is much easier to filter both spatially and temporally.
  • Fat ray payloads / long ray extents — for occlusion-type queries (shadows, AO), use accept_any_intersection(true) so traversal stops at the first hit instead of finding the closest one, and cap max_distance aggressively (AO rays rarely need to travel more than a meter or two).
  • Divergent traversal — rays within one SIMD-group should be coherent when possible. Screen-space-ordered dispatch is already decent; avoid randomly permuting pixel-to-thread assignment.

Anatomy of a denoised RT effect

Any stochastic RT effect (AO, shadows from area lights, diffuse GI, reflections) converges to the same pass skeleton. The insight is that the ray tracing pass is the cheap part — you deliberately undersample it and spend the savings on filtering:

G-buffer (depth, normal, motion vectors) │ ▼ [1] RayGen 1–2 rays/px, blue noise, any-hit, short rays │ → noisy scalar/color target ▼ [2] Temporal accumulate reproject with motion vectors, │ blend with history (α ≈ 0.05–0.2), │ clamp/reject on disocclusion ▼ [3] Spatial filter edge-aware blur (à-trous bilateral), │ 2–3 widening iterations, │ edge stops on depth + normal ▼ [4] Composite apply to ambient/indirect terms only

Notes on each stage:

  • RayGen writes to a dedicated RT (for AO: single-channel R8Unorm or R16Float). Optionally half resolution.
  • Temporal accumulation needs motion vectors (or camera-only reprojection using previous frame's view-projection and depth — sufficient for static scenes) plus the previous frame's result and depth. Disocclusion detection compares reprojected depth/normal against history; on mismatch, reset the history weight rather than ghosting.
  • Spatial filter: the à-trous wavelet form samples a 5×5 kernel with a stride that doubles per iteration (1, 2, 4), giving a large effective radius for 3 × 25 taps. Bilateral weights from depth and normal differences keep edges crisp. This is the SVGF family recipe, minus the variance guidance — variance can be added later.
  • Composite: AO should attenuate ambient/IBL and diffuse indirect lighting only. Multiplying direct sunlight by AO is physically wrong and visibly dirties lit surfaces.

In this engine, each stage maps to one RenderPass subclass in the render graph, so every stage shows up individually in the GPU timing panel — which is exactly what the panel was built for.

Case study: the road to real RT AO

Current state: RaytraceAOPass dispatches 3d_ssao.metal, which is a 32-sample screen-space AO — it marches the depth buffer and never issues a ray. The host binds the TLAS at buffer index 2, but the kernel does not declare it; the binding is dead. SSAO's characteristic failure modes (halos at silhouettes, missing occlusion from off-screen or depth-occluded geometry) apply.

Implementation plan, in commit-sized steps:

Step 1 — replace the kernel with a true ray-traced version

New shader 3d_raytrace_ao.metal, reusing the world-position/normal reconstruction already present in 3d_raytrace_shadow.metal:

#include <metal_stdlib>
#include <metal_raytracing>
using namespace metal;
using namespace metal::raytracing;

constant uint  RAYS_PER_PIXEL = 2;      // temporal accumulation supplies the rest
constant float AO_RAY_LENGTH  = 1.5;    // meters; also the falloff range

kernel void computeMain(/* depth, normal, aoOut, FrameData, CameraData, */
                        instance_acceleration_structure TLAS [[buffer(2)]],
                        uint2 tid [[thread_position_in_grid]]) {
    // reconstruct worldPos, worldNormal from depth + normal RTs (as today)

    intersector<instancing> isect;
    isect.accept_any_intersection(true);   // occlusion query: first hit wins
    isect.assume_geometry_type(geometry_type::triangle);

    float occlusion = 0.0;
    for (uint i = 0; i < RAYS_PER_PIXEL; i++) {
        float2 xi  = blueNoise(tid, frame.frameNumber, i);   // not white noise
        float3 dir = cosineSampleHemisphere(xi, worldNormal);

        ray r;
        r.origin       = worldPos + worldNormal * 1e-3;
        r.direction    = dir;
        r.min_distance = 0.0;
        r.max_distance = AO_RAY_LENGTH;

        auto hit = isect.intersect(r, TLAS);
        if (hit.type != intersection_type::none) {
            // distance falloff: nearby occluders matter more
            occlusion += 1.0 - saturate(hit.distance / AO_RAY_LENGTH);
        }
    }
    aoOut.write(1.0 - occlusion / float(RAYS_PER_PIXEL), tid);
}

Key differences from the shadow kernel: cosine-weighted hemisphere directions (importance-samples the AO integrand), short capped rays, and any-hit acceptance. With 2 rays/px this pass should land well under the current 6.6 ms — the rays are short and coherent.

Step 2 — temporal accumulation pass

New AOTemporalPass (compute) with new resources: aoHistoryRT (ping-pong pair) and previous-frame depth. Until the engine has per-object motion vectors, camera-only reprojection is enough:

float2 prevUV   = project(prevViewProj, worldPos);      // reproject
float  prevAO   = aoHistory.sample(s, prevUV).r;
float  prevDepth= prevDepthTex.sample(s, prevUV).r;
bool   valid    = all(prevUV == saturate(prevUV))
               && abs(linearize(prevDepth) - linearize(expectedDepth)) < eps;
float  alpha    = valid ? 0.1 : 1.0;                    // reset on disocclusion
out = mix(prevAO, currentAO, alpha);

This multiplies effective sample count by ~10× for one cheap full-screen pass.

Step 3 — spatial à-trous filter

New AODenoisePass running 2–3 iterations of a 5×5 bilateral kernel with doubling stride, edge-stopping on depth and normal:

w = gaussian[tap]
  * exp(-abs(depthC - depthT) / sigmaD)
  * pow(max(0.0, dot(normalC, normalT)), sigmaN);

Iterations ping-pong between aoRT and a scratch RT. Each iteration is a trivial compute pass — three more rows in the timing panel.

Step 4 — composite correctly

Today aoRT is consumed in PostProcessPass as a whole-image multiply. Move the AO application into the lighting path so it attenuates only the ambient/IBL contribution, not direct light.

Expected budget

PassTarget (M3, 1440p)
RayGen (2 rays/px, short rays)~1.5–2.5 ms
Temporal accumulate~0.2 ms
Spatial filter ×3~0.6 ms
Total~2.5–3.5 ms, vs 6.6 ms today — with dramatically better quality (no halos, off-screen occlusion, temporal stability)

If the budget needs to shrink further, the first lever is half-resolution raygen + edge-aware upsample in the composite (roughly halves the total again).

Consequences

  • Four dispatch call sites converted from 1×1 threadgroups to dispatchThreads with 8×8 groups; 3d_tile_light_cull.metal now indexes by thread position.
  • Measured on M3: no significant change for the RT shadow pass — it is traversal-bound on RT hardware, not issue-bound. The fix is kept as correct-by-default practice; the expected wins are on ALU-bound kernels and on M1/M2 software traversal. The shadow pass's actual levers are any-hit acceptance, payload reduction, and ray extent (see above).
  • True RT AO is specced as a four-step plan (raygen → temporal → spatial → composite) with new resources: aoHistoryRT ping-pong pair, previous-frame depth, blue-noise texture.
  • TLAS refit/skip remains an open TODO, tracked in TLASBuildPass.
