3. Frame Lifecycle

SDL_PollEvents()
      ├─→ InputManager::processEvent()    // keyboard/mouse → InputState
      └─→ EngineCore::processRmlUIEvent() // UI consumes first

EngineCore::update(dt)
      ├─→ ActionManager::update(dt)       // advance tweens/timers
      ├─→ AudioEngine::update(dt)        // fire completion callbacks
      └─→ ResourceManager                 // async tasks on TaskScheduler

Physics3D::process(scene, dt)            // fixed 60Hz step
      ├─→ JPH::JobSystemThreadPool        // broadphase + narrowphase
      ├─→ ContactListener callbacks       // onCollisionEnter/Exit
      ├─→ TriggerListener callbacks       // onTriggerEnter/Exit
      └─→ write back Node::localTransform

Scene::update(dt)
      └─→ recursive world transform update (dirty nodes only)

Renderer::draw(scene, camera)
      ├─→ stage geometry to GPU (if dirty)
      ├─→ depth pre-pass
      ├─→ cluster light culling compute
      ├─→ PBR forward pass (instanced)
      ├─→ sky / atmosphere
      ├─→ water pass
      ├─→ volumetric fog / clouds / god rays
      ├─→ post-process: bloom → DOF → tone mapping
      ├─→ 2D/3D batch flush
      ├─→ UI pass (RmlUI)
      └─→ debug wireframe (optional)

4. Key Abstractions

4.1 Renderer Interface

Renderer is a pure virtual interface. Metal and Vulkan each provide a full implementation, selected at runtime via createRenderer(GraphicsBackend). Engine code only ever holds a Renderer*.

This is the most brittle boundary: renderer_metal.hpp exposes 31 concrete render pass types. Any pass change requires both backends to stay in sync. See the parity matrix for current backend divergence.

4.1a Graphics Header Modularization

The original graphics.hpp (544 LOC god header) was split into five focused sub-headers:

HeaderContents
graphics_handles.hppType-safe GPU handles via GPUHandle<Tag> template
graphics_batch2d.hppBlendMode, Batch2DVertex, Batch2DStats
graphics_gpu_structs.hppGPU layout: MaterialData, lights, CameraData, InstanceData, clusters
graphics_mesh.hppCPU-side: VertexData, Mesh, Material, Image
graphics_effects.hppEffect structs: water, atmosphere, fog, clouds, sun flare, GPU particles

graphics.hpp is kept as a 16-LOC umbrella include for full backwards compatibility.

4.2 Node Scene Graph

Node uses a shared_ptr tree. localTransform is set by the user; worldTransform is computed lazily in Scene::update() via isTransformDirty. Physics body positions are read back from Jolt each frame and written into Node::localTransform — one-way data flow.

4.3 Resource<T>

A future-like handle. Starts as Loading, transitions to Ready. get() blocks; tryGet() does not. ResourceCache<T> deduplicates by path across threads.

4.4 Handle System

Physics bodies, triggers, textures, fonts, and audio sources use uint32_t-wrapped handles (BodyHandle, TriggerHandle, TextureHandle…). UINT32_MAX is the sentinel for invalid. GPU resource handles gained compile-time type safety via GPUHandle<Tag>; physics handles are still untyped.

6. Known Weaknesses

6.1 Vaporware/src/main.cpp

All game initialization, scene construction, game loop logic, camera control, and UI animation live in a single file. This is a demo problem, not an engine problem — but it makes the correct usage of the engine invisible. Integration tests are not currently possible.

6.2 Renderer Pass Count

renderer_metal.hpp forward-declares 31 render pass structs with no shared interface — coupling is implicit via call order. Vulkan currently implements 3 passes to Metal's 31. Vulkan is a second-class backend.

6.3 graphics.hpp God Header (Fixed 2026-05-01)

Resolved by splitting into 5 sub-headers. See §4.1a.

6.4 Untyped Physics Handles

BodyHandle and TriggerHandle are both bare uint32_t wrappers. Mixing them is a naming-convention bug, not a compile error. GPU handles have been fixed; physics handles have not.

6.5 Dual Object Model

The demo uses EnTT ECS (entt::entity); the engine uses a shared_ptr Node tree. Both coexist with no unified mapping. A scene object can be either a Node* or an entt::entity depending on which layer you're in.

8. Strengths

StrengthNotes
Library / app separationVapor does not depend on Vaporware. Can be used as a standalone library.
Graphics API abstractionRenderer interface isolates Metal/Vulkan from user code.
Thread-safe resource loadingResourceManager + TaskScheduler designed correctly for safe async loading.
Physics / scene separationPhysics3D is an independent singleton. Interaction is driven via process(scene).
Handle-based APINo raw pointers in the public API. Internals can be swapped without breaking callers.
Type-safe GPU handlesGPUHandle<Tag> provides compile-time handle type safety in the graphics layer.
Shape cacheSame-dimension shapes are shared automatically, avoiding redundant Jolt allocations.
CICatch2 v3 + GitHub Actions. 76 tests run on every push (macos-15, Ninja, ccache).
