1. Frustum Culling 的現行實作方式

1.1 執行位置

在目前的 Metal 後端（renderer_metal.cpp）裡， Frustum Culling 發生在 渲染 Pass 的 draw 階段， 而不是在 Instance 資料上傳 GPU 之前。 具體位置是兩個地方：

呼叫點場景對應 Pass
camera.hpp:476 Per-mesh AABB 球體測試，在 Pre-pass（深度預測）裡 Pre-pass / Shadow
camera.hpp:993 Per-mesh AABB 球體測試，在主要 Lighting Pass 裡 Forward Lighting

兩者都呼叫 Camera::isVisible(AABB)， 該函數把 AABB 的最小/最大角投影到六個 Frustum Plane 做保守 AABB 測試。 判定為不可見時，直接跳過 draw call 而不是從 Instance 陣列移除。

1.2 boundingSphere 欄位從未填值

已知缺陷
InstanceData 結構體有一個 boundingSphere 欄位（glm::vec4，xyz = center, w = radius）， 但整個 codebase 中沒有任何地方在上傳 GPU 前填入這個值。 目前 GPU 端 Light Culling Compute Shader 若需要它，行為將是未定義。

1.3 GPU Culling 的邊界

GPU 側的 Tile/Cluster Light Culling（tileCullingPipeline）只剔除 光源（哪些 tile 被哪些點光源影響）， 並 不 對幾何體做 Occlusion 或 Frustum Culling。 幾何層級的 Culling 目前完全是 CPU 責任。

為什麼 Culling 放在 Pass 內，而不是 Instance 上傳前？
目前 Metal 後端在 stage() 階段就把所有 Instance 資料寫進 GPU Buffer， draw() 階段才知道 Camera 的位置（Camera 可能每幀移動）。 因此 Culling 只能在 draw loop 裡做，缺點是已有 GPU memory 開銷。 ECS 化後可以把 Camera 提前，在上傳前做 Culling，大幅減少頻寬。

1.4 Vulkan 後端現況

renderer_vulkan.cpp 的 draw() 函數目前尚未實作完整的 Instance 逐個 Culling，Batch2D 的 draw call 也只有存根（stub）。 Frustum Culling 是 ECS 重構後第一個要補齊的渲染功能。

2. Frustum 計算與 Transform 計算的關係

2.1 兩條獨立的 Dirty-Flag 鏈

Frustum 更新與 Transform 傳播是完全獨立的兩條鏈， 只在 Camera::isVisible() 呼叫時交會。

【Transform 鏈】
  Node::localTransform 改變
    → Node::isTransformDirty = true
    → Scene::update() 逐節點 DFS 傳播
    → worldTransform 重算（父矩陣 × 本地矩陣）
    → World AABB 重算（mesh->getLocalAABB() × worldTransform）

【Frustum 鏈】
  Camera::setEye / setCenter / setFov
    → _isFrustumDirty = true
    → 下次呼叫 getFrustumPlanes() 時 lazy 重算
        （proj × view 矩陣 → 6 平面 Gribb-Hartmann 提取）

【交會點】
  draw() 裡： camera.isVisible(node->worldAABB)

2.2 ECS 分支的改動

在 ecs-refactor 分支，TransformComponent 的結構已由 單一 glm::mat4 localTransform 改為分解形式： 

struct TransformComponent {
    glm::vec3  position;
    glm::quat  rotation;
    glm::vec3  scale;
    glm::mat4  worldTransform;   // 快取，由 system 更新
    entt::entity parent;
    bool       isDirty;
};

這意味著 Dirty 旗標仍然存在，但傳播機制從 Node 樹的 DFS 改為 EnTT registry.view<TransformComponent>() 的線性掃描。 在 ECS 架構下，Transform System 負責在渲染前把所有 isDirty == true 的實體重算 worldTransform， Frustum System（或仍在 Camera 側）再做可見性測試。

2.3 更新順序要求

系統執行順序
每幀必須嚴格依序：
TransformSystem → FrustumCullSystem → UploadInstanceSystem → RenderSystem

若 FrustumCullSystem 先於 TransformSystem 執行， 使用的是上一幀的 worldTransform 做裁剪，結果將是一幀延遲的錯誤剔除。
Frustum Plane 本身需要 dirty flag 嗎？
camera.hpp 中已有 _isFrustumDirty， 並在 setEye / setCenter / setFov / setPerspective 被設為 true， getFrustumPlanes() 呼叫時 lazy 重算。 ECS 化後建議把 Camera 本身也做成 Component（VirtualCameraComponent 已存在於 components.hpp）， 讓 Camera System 統一處理 Frustum 更新，避免散落在各處的手動呼叫。

3. ECS 化後渲染 Loop 的形狀

3.1 現行 Node Tree 的遍歷（Before）

// renderer_metal.cpp — 現行方式（偽代碼）
void Renderer::buildInstances(Scene* scene) {
    scene->traverseNodes([&](Node* node) {   // DFS，O(n) 但有 cache miss
        for (auto& mesh : node->meshes) {
            instances.push_back({
                node->worldTransform,
                mesh->materialID,
                /* boundingSphere 從未填 */
            });
        }
    });
    // 上傳整批 Instance，Culling 在 draw() 時做
}

3.2 ECS 後的預期形狀（After）

// 目標架構 — ECS 版本（偽代碼）

// System 1: Transform（必須最先）
auto dirtyView = registry.view<TransformComponent>();
for (auto e : dirtyView) {
    auto& t = dirtyView.get<TransformComponent>(e);
    if (t.isDirty) recomputeWorldTransform(registry, e, t);
}

// System 2: Frustum Cull（在上傳前完成，節省頻寬）
auto meshView = registry.view<TransformComponent, MeshRendererComponent>();
std::vector<InstanceData> visible;
for (auto e : meshView) {
    const auto& [tc, mc] = meshView.get(e);
    if (!mc.visible) continue;
    AABB worldAABB = transformAABB(mc.localAABB, tc.worldTransform);
    if (!camera.isVisible(worldAABB)) continue;    // ← Culling 在此
    for (auto& mesh : mc.meshes) {
        visible.push_back({tc.worldTransform, worldAABB, mesh->materialID});
    }
}

// System 3: 上傳（只上傳可見的）
uploadInstanceBuffer(visible);

// System 4: Render（不再做 per-draw culling）
for (auto& inst : visible) {
    bindMaterial(inst.materialID);
    drawIndexed(...);
}

3.3 關鍵差異對比

面向Node Tree（現行）ECS（目標）
遍歷方式 遞迴 DFS，pointer chasing 線性 view 掃描，cache friendly
Frustum Culling 時機 draw() 裡，per-draw call 上傳前，CPU 批次篩選
Instance 上傳量 全部 instance（含不可見） 只有通過 culling 的 instance
Camera 存取 draw() 時傳入 作為 Component，在 System 間共享
並行潛力 低（DFS 依賴父子關係） 高（各 archetype 可平行處理）
boundingSphere 從未填值 在 MeshRendererComponent 載入時計算
設計建議
EnTT 的 registry.view<A, B>() 只迭代同時擁有兩個 Component 的實體， 不需要手動過濾。搭配 entt::exclude<DisabledTag> 可以零開銷地跳過停用的實體，取代現行的 if (!mc.visible) continue。

4. Migration 過程最容易出錯的地方（風險排序）

4.1 風險總覽

排名問題風險相關位置
#1 Transform 更新時機不對，Culling 用到舊數據 高 TransformComponent::isDirty
#2 Node 與 ECS Component 雙重更新導致數據不一致 高 scene.hpp, Node vs TransformComponent
#3 Instance ID 不穩定造成 Material Descriptor 對應錯誤 中 renderer_vulkan.cpp: materialIDs
#4 boundingSphere 欄位空值，GPU Light Culling 結果錯誤 中 graphics.hpp: InstanceData
#5 父子 Transform 傳播順序在線性掃描下未保證 低 TransformComponent::parent

4.2 #1 — Transform 更新時機

問題：如果 TransformSystem 的執行時機排在 Culling 之後， Culling 讀到的 worldTransform 是上一幀的值。 快速移動的物件（玩家角色、拋射物）在低幀率下可能被錯誤剔除， 產生閃爍或消失的視覺 bug。

防範：明確定義系統執行順序清單，並加入 assert(transformSystemRanThisFrame) 這類 debug guard。

最高風險
這類 bug 在靜態場景中完全不會出現，只有在相機快速移動 或物件快速移動時才會觸發，非常難以重現和追蹤。

4.3 #2 — Node 與 ECS 雙重更新

問題：目前 ecs-refactor 分支的 scene.hpp 仍保有 Node 結構，同時 components.hpp 有獨立的 TransformComponent。在過渡期間，如果有程式碼同時寫入 node->localTransform 和 TransformComponent::position， 兩套數據將會分歧，渲染出來的位置與物理/邏輯位置不一致。

防範：選定一個「唯一真實來源」（single source of truth）。 過渡期可用橋接 Component（如 SceneNodeReferenceComponent） 讓 ECS System 自動同步 Node，但必須有明確的同步點，不能雙向寫。

4.4 #3 — Instance ID 穩定性

問題：Vulkan 後端的 materialIDs Map 以 std::shared_ptr<Material> 為 key 分配遞增 ID。 ECS 重構後，Material 的生命週期由 Component 管理， 如果 shared_ptr 被替換（即使指向同一份資料）， ID 就會重新分配，導致 VkDescriptorSet 對應關係錯亂。

防範：將 Material ID 改為穩定的 entt::entity 或 UUID，不依賴 pointer 值。

4.5 #4 — boundingSphere 空值

問題：InstanceData::boundingSphere（glm::vec4） 在現行 codebase 中沒有任何地方被填值就上傳 GPU。 ECS 重構後若 GPU Compute Shader 開始依賴這個欄位做 Occlusion Culling， 將產生全黑畫面或 GPU validation 報錯。

防範：在 MeshRendererComponent 建立時， 從 mesh->getLocalAABB() 計算出包圍球（取 AABB 中心 + 對角線長度的一半）， 存入 Component，在 Upload System 時寫入 InstanceData。

4.6 #5 — 父子 Transform 傳播順序

問題：線性 registry.view() 迭代的順序不保證父節點比子節點先被處理。 如果子節點的 Transform 比父節點更早更新，子節點的 worldTransform 將乘上父節點上一幀的矩陣，產生一幀延遲的位移。

防範：在 TransformSystem 中，使用 BFS 或拓撲排序 確保父節點優先，或者用迭代直到收斂（允許多幀傳播，但有額外開銷）。 EnTT 的 entt::forest 或自定義 Children Component 可以輔助有序遍歷。

完全移除 Node Tree 後，Scene Graph 的層級關係由誰維護？
由 TransformComponent::parent（entt::entity）維護。 這等同於把 Node 的父子指標拆成 Component， 每個實體只需記住自己的父實體 ID， TransformSystem 在更新時向上遞迴或以拓撲順序展開。 這種方式沒有 Node 的 DFS pointer chasing， 但需要顯式處理「孤兒」（parent = entt::null）和循環參照防護。
ECS 化後，Batch2D（UI 覆蓋層）需要怎麼整合？
Batch2D 本質上是螢幕空間的即時提交（immediate mode）， 不需要 Transform 傳播或 Frustum Culling。 建議保持獨立的 Batch2DSystem， 在 RenderSystem 之後執行，直接呼叫 renderer->drawQuad2D()。 UI Component（如 SpriteComponent、TextComponent） 只需攜帶螢幕座標或 NDC 座標，不需要加入主要的 Instance Upload 流程。
